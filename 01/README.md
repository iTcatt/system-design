# Домашнее задание 01: Сервис доставки (Вариант №6)


## Описание варианта

**Вариант №6 — Сервис доставки www.sdek.ru**

Система для управления посылками (items) и доставками между пользователями.

### Предметная область

| Сущность | Описание |
|----------|----------|
| **Пользователь (User)** | Отправитель или получатель предмета |
| **Предмет (Item)** | Объект, который необходимо доставить (может иметь разные характеристики в зависимости от типа) |
| **Доставка (Delivery)** | Процесс перемещения предмета от отправителя к получателю |

### API

| Метод | Endpoint | Описание |
|-------|----------|----------|
| `POST` | `/api/user` | Создание нового пользователя |
| `GET` | `/api/user?login={login}` | Поиск пользователя по логину |
| `GET` | `/api/user?name={mask}` | Поиск пользователя по маске имени и фамилии |
| `POST` | `/api/item` | Создание посылки |
| `GET` | `/api/item?userId={id}` | Получение всех посылок пользователя |
| `POST` | `/api/delivery` | Создание доставки от пользователя к пользователю |
| `GET` | `/api/delivery?receiverId={id}` | Получение информации о доставке по получателю |
| `GET` | `/api/delivery?senderId={id}` | Получение информации о доставке по отправителю |

---

## Роли пользователей

| Роль | Описание |
|------|----------|
| **Отправитель** | Создаёт предметы, инициирует доставки, отслеживает статус отправлений |
| **Получатель** | Отслеживает доставки, просматривает информацию о предметах |
| **Администратор** | Управляет пользователями, осуществляет поиск по логину и маске имени |

---

## Внешние системы

| Система | Назначение |
|---------|-----------|
| **Email/SMS Сервис** | Отправка уведомлений о статусе доставки |
| **Платёжная система** | Обработка оплаты за доставку |
| **Геолокационный сервис** | Отслеживание текущего местоположения предмета по трек-номеру |


## Диаграммы

### 1. System Context (C1)
Общий контекст системы: роли пользователей, основная система и внешние системы.

### 2. Container (C2)
Внутренняя структура системы: микросервисы, базы данных, брокер сообщений и их взаимодействие.

### 3. Dynamic — Создание доставки
Архитектурно значимый сценарий: **создание доставки от отправителя к получателю с оплатой**.

Последовательность взаимодействия:
1. Отправитель заполняет форму создания доставки
2. Web Application отправляет `POST /api/delivery` на API Gateway
3. API Gateway передаёт запрос в Delivery Service
4. Delivery Service проверяет отправителя через User Service
5. Delivery Service проверяет получателя через User Service
6. Delivery Service проверяет предмет через Item Service
7. Delivery Service инициирует оплату через Платёжную систему
8. Delivery Service сохраняет запись о доставке в Delivery Database
9. Delivery Service публикует событие `DeliveryCreated` в Message Broker
10. Notification Service получает событие из Message Broker
11. Notification Service отправляет уведомление получателю через Email/SMS Сервис

### 4. Dynamic — Отслеживание предмета
Сценарий: **получатель отслеживает местоположение предмета**.

Последовательность взаимодействия:
1. Получатель открывает страницу отслеживания
2. Web Application отправляет `GET /api/delivery/{id}/tracking` на API Gateway
3. API Gateway передаёт запрос в Delivery Service
4. Delivery Service получает трек-номер доставки из Delivery Database
5. Delivery Service запрашивает геолокацию по трек-номеру у Геолокационного сервиса
6. Delivery Service возвращает координаты через API Gateway
7. API Gateway передаёт данные в Web Application
8. Web Application отображает карту с текущим положением предмета
